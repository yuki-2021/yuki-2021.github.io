在数据比较少时候，表中数据可以完全缓存到内存，全表扫描也不会太慢。

数据越来越多，不能完全缓存，索引作用越来越明显。

Mysql中有Btree和Hash索引。不同的存储引擎对于索引的实现方式可以能不同

BeTree索引

  • 数据解构是B+树。对于InnoDB，指针指向的是主键索引位置。对于MyIsaM,索引指向的是数据的物理地址。
  • 使用索引，查找合适范围的数据，根据指针，找到主键，根据主键找到物理地址，只需要加载少量内容。
  • 适合范围查找，数据顺序存储。

Betree使用

单列索引

  • 精确匹配、模糊前缀匹配、范围查询(between and 、>)
  • Not in 和!= 无法用索引

多列索引

  • 最左前缀匹配
  • 前面的列只有是精切匹配，后面的列才会生效
  • 范围查询和模糊匹配的右边列无法生效

Betree使用限制

  • 使用索引，命中了大部分的数据，可能会导致采用全表扫描，而不是索引。

  • 不是最左匹配，那么无法使用索引。

Hash索引

  • 在InnoDB中，不能自己建立，InnoDB根据Betree的使用情况，自动建立Hash索引。
  • Hash索引中存储的是Hash码和指针。
  • Hash索引只能用于精确匹配。
  • Hash索引需要两次读取，先找到行，然后对行进行读取。
  • 无法用于排序
  • 只能全键值匹配，
  • Hash索引可能导致Hash冲突 - 不适合选择性差的列上

索引好处

  • 减少需要扫描数据的数据量。数据存储在页中，一页数据越多，效率越高。
  • 可以使用索引排序，避免了用临时表排序。
  • 可以把I/O编程顺序IO。

索引越多越好

  • 索引会导致写成本。
  • 索引会增加查询优化器的选择时间。

索引优化策略

  • 索引列上不能使用函数表达

  • Betree索引大小有限制，对于InnoDB，最大是700+Byte。对于MyIsam是1000Byte。对于Text等大文本，建立前缀索引。

  • 5.0之前，只能用单列索引。5.0之后，有了索引合并的概念。可以使用联合索引。可以 - 1. 经常用于查询的列优先 2. 选择性高的列优先。3.宽度小的列优先

  • 覆盖索引。直接在Betree索引上获取数据，而不需要读取表。全不知，不止是where包括select和groupby和orderby中的值。好处 1. - 可以优化缓存，减少磁盘IO操作。2. - 减少随机IO，变随机IO为顺序IO. 3. - 避免对InoDB主键索引的二次查询 4. - 可以避免MyISAM，避免系统带哦用。 无法用 - 1. 不都支持覆盖索引，Memeory。2. hash不能当作覆盖索引 3. 查询太多列，没必要建立覆盖索引 3. - 不能有like%%的查询 using index - using where

    查询优化中

    优化排序 - using index

    按照索引顺序扫描数据，进行排序。

    • 索引顺序和Order by 子句顺序完全一直
    • 索引排序顺序和asc、desc完全一直
    • 关联查询，orderby字段在第一个表中

    模拟Hash索引

索引优化下

  • Inodb行锁。使用索不到行，对行加锁，在内存中过滤。过滤到行，减少锁定行数。
  • 加快处理速度，加快锁的释放熟读
  • 重复索引
  • primary key(id),unique key(id),index(id)
  • index(a),index(a,b)
  • primary key(id),index(a,id)
  • 删除重复和冗余索引
  • pt-duplicate-key-check h = 127.0.0.1

Spring概述

Spring是一个用于组件管理的容器。

  • 我们把配置信息提供给Spring，Spring把配置信息转换成BeanDefinition，帮我们完成依赖注入。这个过程叫做IOC。

  • Spring支持AOP。我们把横切逻辑放到Advice，Spring在创建Bean的时候，会采用gdk动态代理或者cglib方式，给我们返回代理类。

  • Spring支持组件扫描和自动注入和自动代理。我们不再需要自己提供配置信息，Spring自动完成组件扫描和依赖注入DI。

  • 使用Spring可以整合ORM、aspectj、spring mvc、Spring Transition和J2ee技术。

  • Spring程序支持多种单元测试框架，比如Junit，可以容易的获取到ApplicationContext。

数据库选型思路

虽然数据库技术选型不是自己的工作，这里还是记录下笔记。

主要思路如下

  • 确定数据库类型 - 关系型数据库(SQL)和非关系型数据库(NOSQL)
  • 确定数据产品

数据库定型

数据库特点

关系型数据库特点如下

  • 表结构预先定义 - 适合列固定的数据
  • 支持事务 - 保证数据完整性
  • 使用SQL - 支持复制查询

NOSQL以Mongo为例，特点如下

  • 数据列不固定
  • 不支持事务 - 并发能力高
  • 数据操作比较简单的简单
根据业务选数据库

接着，根据业务选择合适的数据库。

  • 比如，用来存储苹果的特点，这些特点可以动态变化，这就要用NOSQL。
  • 比如，你要做日志存储和用户行为分析，一两条数据丢失，不会对总体产生影响，这时候纪要用NOSQL。
  • 比如，你要存储财务数据，这时就要用SQL。

产品定型

产品定型考虑因素

  • 数据库可用性(可用率达到多少)、安全性(数据库安全控制怎么样)、扩展性(是不是能换存储引擎、插件多少)
  • 考虑是否免费和开发人员成本(开发人员是否容易招聘)
  • 广泛性(使用率如何、是否容易找相关文章)

对于MySQL来说

  • Mysql主从复制集群能达到99%可用，对于高可用架构配合主从复制可以达到99.99%可用。
  • Mysql支持列级别的安全控制
  • Mysql支持众多的数据库中间件(和业务无关的通用插件),支持读写分离和分库分表。
  • Mysql支持二进制的日志逻辑复制
