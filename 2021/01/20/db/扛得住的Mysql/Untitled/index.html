在数据比较少时候，表中数据可以完全缓存到内存，全表扫描也不会太慢。

数据越来越多，不能完全缓存，索引作用越来越明显。

Mysql中有Btree和Hash索引。不同的存储引擎对于索引的实现方式可以能不同

BeTree索引

  • 数据解构是B+树。对于InnoDB，指针指向的是主键索引位置。对于MyIsaM,索引指向的是数据的物理地址。
  • 使用索引，查找合适范围的数据，根据指针，找到主键，根据主键找到物理地址，只需要加载少量内容。
  • 适合范围查找，数据顺序存储。

Betree使用

单列索引

  • 精确匹配、模糊前缀匹配、范围查询(between and 、>)
  • Not in 和!= 无法用索引

多列索引

  • 最左前缀匹配
  • 前面的列只有是精切匹配，后面的列才会生效
  • 范围查询和模糊匹配的右边列无法生效

Betree使用限制

  • 使用索引，命中了大部分的数据，可能会导致采用全表扫描，而不是索引。

  • 不是最左匹配，那么无法使用索引。

Hash索引

  • 在InnoDB中，不能自己建立，InnoDB根据Betree的使用情况，自动建立Hash索引。
  • Hash索引中存储的是Hash码和指针。
  • Hash索引只能用于精确匹配。
  • Hash索引需要两次读取，先找到行，然后对行进行读取。
  • 无法用于排序
  • 只能全键值匹配，
  • Hash索引可能导致Hash冲突 - 不适合选择性差的列上

索引好处

  • 减少需要扫描数据的数据量。数据存储在页中，一页数据越多，效率越高。
  • 可以使用索引排序，避免了用临时表排序。
  • 可以把I/O编程顺序IO。

索引越多越好

  • 索引会导致写成本。
  • 索引会增加查询优化器的选择时间。

索引优化策略

  • 索引列上不能使用函数表达

  • Betree索引大小有限制，对于InnoDB，最大是700+Byte。对于MyIsam是1000Byte。对于Text等大文本，建立前缀索引。

  • 5.0之前，只能用单列索引。5.0之后，有了索引合并的概念。可以使用联合索引。可以 - 1. 经常用于查询的列优先 2. 选择性高的列优先。3.宽度小的列优先

  • 覆盖索引。直接在Betree索引上获取数据，而不需要读取表。全不知，不止是where包括select和groupby和orderby中的值。好处 1. - 可以优化缓存，减少磁盘IO操作。2. - 减少随机IO，变随机IO为顺序IO. 3. - 避免对InoDB主键索引的二次查询 4. - 可以避免MyISAM，避免系统带哦用。 无法用 - 1. 不都支持覆盖索引，Memeory。2. hash不能当作覆盖索引 3. 查询太多列，没必要建立覆盖索引 3. - 不能有like%%的查询 using index - using where

索引优化下

  • Inodb行锁。使用索不到行，对行加锁，在内存中过滤。过滤到行，减少锁定行数。
  • 加快处理速度，加快锁的释放熟读
  • 重复索引
  • primary key(id),unique key(id),index(id)
  • index(a),index(a,b)
  • primary key(id),index(a,id)
  • 删除重复和冗余索引
  • pt-duplicate-key-check h = 127.0.0.1
